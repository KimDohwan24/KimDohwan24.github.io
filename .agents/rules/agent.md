- **패키지 매니저:** `pnpm`을 사용하십시오. (`npm`이나 `yarn` 사용 금지)

## 2. Git 및 버전 관리 (Version Control)
- **사용자 정보:** 시스템 전역(`git config --global`)에 설정된 Username과 Email을 그대로 사용하십시오.

### 2.1 브랜치 전략 (Branching Strategy)
- **Github Flow 준수:**
  - 절대로 `main` 브랜치에 직접 커밋(Commit)하지 마십시오.
  - 작업 유형에 따라 적절한 접두사를 가진 새 브랜치를 생성하여 작업하십시오.
    - 예: `feat/login-page`, `fix/navbar-bug`, `chore/setup-docker`
  - **병합(Merge) 규칙:** 브랜치 병합 시 반드시 **Squash Merge**를 사용하십시오. 작업 브랜치의 커밋을 하나로 합쳐서 병합 이력을 단순하게 유지합니다.
    - 명령어 예시: `git checkout main && git merge --squash feature-branch && git commit`

### 2.2 커밋 메시지 (Commit Messages)
- **언어:** 커밋 메시지는 **한국어**로 작성하십시오.
- **형식:** [Conventional Commits](https://www.conventionalcommits.org/) 형식을 따르십시오.
  - `feat: 새로운 기능 추가`
  - `fix: 버그 수정`
  - `chore: 빌드 업무 수정, 패키지 매니저 설정 등`
  - `refactor: 코드 리팩토링`
  - `docs: 문서 수정`

## 4. 포트폴리오 설명 작성 원칙 (5대 원칙)
포트폴리오의 프로젝트 설명과 하이라이트를 작성 및 수정할 때는 다음 5가지 원칙을 반드시 반영하여 단순 기능 나열이 아닌 의사결정과 문제 해결력이 돋보이도록 하십시오.

1. **프로젝트를 보는 법 (문제 정의 중심)**
   - 단순히 만든 기능만 나열하지 말고, **어떤 비즈니스적/기술적 문제를 해결하기 위해 이 프로젝트를 시작했는지**를 전면에 배치하십시오.
2. **기능 설명 (판단과 의사결정 중심)**
   - "캘린더 구현", "알림 구현"처럼 단순히 한 일(Action)만 적기보다는, **어떤 불편함이 있어 해당 기능에 우선순위를 두고 어떤 판단 하에 구현했는지**가 보이도록 하십시오.
3. **기술 스택 한 줄 (선택 사유 명시)**
   - 단순히 사용한 기술명만 적지 말고, **"조회 수가 많은 인기글 조회 속도를 높이기 위해 Redis 캐싱 도입", "확장성을 고려한 JWT 설계"**와 같이 해당 기술을 선택한 타당한 이유와 트레이드오프를 함께 작성하십시오.
4. **도메인 이해 (Edge Case 고려)**
   - 게시글 등록/조회 같은 기본적인 CRUD에 머무르지 않고, **"신청만 하고 잠수 타는 경우의 예외 처리", "재고 부족 시의 동시성 제어"**와 같이 실제 서비스 운영 중 마주할 수 있는 실질적인 비즈니스 문제를 먼저 짚고 기능에 반영하십시오.
5. **현실 검증 (성능 측정 및 피드백)**
   - 개발 완료에서 끝내지 않고, **실제 배포를 진행하여 사용자 피드백을 수집하거나 부하 테스트(JMeter, nGrinder 등)를 통해 개선 전후의 수치화된 성능지표**를 제시하여 현실적으로 검증하십시오.
